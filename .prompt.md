# Default Prompt for Marching Madness

You are working on **Marching Madness**, a VR marching band simulator in TypeScript + Babylon.js.

## Quick Context

**What**: Players march in formation with an AI band, stepping to beat-synced footstep targets
**Tech**: Babylon.js 7, TypeScript, WebXR, Tone.js, smplr (SoundFont), OSMD (sheet music)
**Status**: Feature-complete with modular architecture; 19 drill formations, auto-march with mouse look

## Key Files to Know

- `src/main.ts` — Core game loop, UI, scene setup
- `src/gameConstants.ts` — Shared constants (BPM=80, scoring, band layout)
- `src/drillMath.ts` — Pure drill math (getDrillPosition, formations, grading)
- `src/drillFactory.ts` — Band creation (createDrill, createCustomDrill)
- `src/audioSystem.ts` — SoundFont loading, spatial audio
- `src/musicManager.ts` — Metronome, Tone.Transport, instrument scheduling from OSMD
- `src/marchingAnimationSystem.ts` — 11 march styles with per-style leg animations
- `tests/unit/basketball.spec.ts` — Unit test suite (2 tests, expand as needed)

## Code Quality Standards

✅ **Must Have Before Commit:**
- 0 TypeScript errors (`npx tsc --noEmit`)
- All tests passing (`npm test`)
- Problems tab clean (no unused vars, missing types)
- Descriptive commit message

❌ **Never**
- Create circular imports
- Duplicate code across modules
- Skip type annotations
- Add logic to main.ts (extract to modules)

## Next Steps (In Order)

1. **Directional Styles** — Select march style based on movement direction vs facing angle
2. **Write Tests** — Add missing drill/scoring tests to `tests/unit/`
3. **Refactor main.ts** — Incrementally wire extracted modules
4. **Missing Systems** — Implement footstepSystem, uiSystem, sheetMusic modules
5. **Performance** — Profile and optimize render loop if needed

## If Something Breaks

1. Check error message carefully (TypeScript is descriptive)
2. Run `npx tsc --noEmit` to locate type issues
3. Look for circular imports (use gameConstants as intermediary)
4. Check that all imports have corresponding exports
5. Verify test coverage if changing core logic

## Questions to Ask

- "What should this module return/export?"
- "Are there tests for this functionality?"
- "Does this belong in gameConstants or a separate module?"
- "Is there a circular dependency lurking?"
- "Have I verified zero errors and all tests pass?"
