# Marching Madness — Claude Project Instructions

## Project Overview

**Marching Madness** is a VR marching band simulator built with Babylon.js 6, TypeScript, WebXR, and Tone.js. Players step to beat-synced footstep targets while managing collisions with AI marchers in a physics-enabled field.

## When Working On This Project

### Always Check First
- Run `npx tsc --noEmit` to verify zero TypeScript errors before committing
- Run `npm test` to ensure all tests pass
- Check that Problems tab is clean (no unused variables, missing types, etc.)

### Module Organization
- **gameConstants.ts** — All numeric/config values (shared constants, no logic)
- **drillMath.ts** — Pure math (formation shapes, position calculations, grading)
- **audioSystem.ts** — Sound generation (Tone.js synths, SoundFont instruments)
- **collisionSystem.ts** — Physics (stumble, domino cascade, particles)
- **environment.ts** — Visual setup (skybox, field)
- **bandMemberFactory.ts** — Mesh creation and instrument assignment
- **firstPersonBody.ts** — Player body animation and locomotion
- **main.ts** — Scene orchestration and render loop

### Code Style
- TypeScript strict mode (non-negotiable)
- Function names are descriptive (`updateCollisions`, `getDrillPosition`, not `update`, `get`)
- Constants are UPPER_CASE
- Interfaces are PascalCase (`StumbleState`, `BandMemberData`)
- Comments explain *why*, not *what*
- Keep functions focused on one responsibility

### Making Changes

#### Adding a Feature
1. **Extract pure logic** to a function (testable, no side effects)
2. **If it's a constant**: Add to `gameConstants.ts`
3. **If it's math**: Add to `drillMath.ts`
4. **If it's a system**: Create a new module (e.g., `footstepSystem.ts`)
5. **Write tests** in `tests/unit/`
6. **Run full suite**: `npm test`
7. **Verify build**: `npx tsc --noEmit`
8. **Commit with clear message**: Include what changed and why

#### Fixing a Bug
1. Write a failing test that reproduces the bug
2. Fix the bug
3. Verify test passes
4. Run full test suite
5. Commit with `[FIX]` prefix

#### Refactoring
- Extract concerns into modules (avoid monolithic files)
- Move duplicated code to reusable functions
- Fix circular dependencies immediately
- Never break existing tests

### Performance Constraints

- **Spatial hash grid** (2.0m cells) for collision O(n) not O(n²)
- **Instanced shadows** (not individual meshes)
- **Dynamic measure generation** (avoid UI stalls)
- **Particle system cleanup** (prevent memory leaks — 6s timeout)
- **Spatial audio weighting** (inverse-distance-squared for fade)

### Quest 3 Specifics

- Test with Firefox/Chrome WebXR emulator first
- Haptic feedback via dual-rumble (if available)
- Controller profiles: standard/standard-triggers
- Hand-tracking filter: skip (grip is unreliable for arm swing)

### Build & Deploy

```bash
npm run build       # TypeScript → JS (Vite)
npm test           # vitest unit tests
npm run dev        # Dev server with HMR
```

Push to `main` branch (GitHub Pages serves from main by default).

### Files to Never Commit

- `.env` (environment variables)
- `node_modules/` (already in .gitignore)
- `dist/` (build output) — though we do commit to GitHub Pages branch
- `*.bak` or `*.tmp` files
- IDE-specific config (VSCode already in .json)

## Common Pitfalls

- ❌ Don't add business logic to main.ts (extract to modules)
- ❌ Don't create circular imports (use gameConstants as intermediary)
- ❌ Don't skip type annotations (strict mode enforces them)
- ❌ Don't commit type errors or unused imports
- ✅ Do extract pure functions (enable testing)
- ✅ Do run tests before committing
- ✅ Do use meaningful variable names
- ✅ Do document non-obvious logic

## Architecture Goals

- **Modularity**: Each file has one job
- **Testability**: Pure functions in gameConstants, drillMath
- **Performance**: Spatial hashing, instancing, dynamic generation
- **Maintainability**: Clear module boundaries, no circular deps
- **Extensibility**: Add features by creating new modules
